## Fases

### Fase 1 · Queries (lib inline no page.tsx)

- `fuelDashboardData(windowDays=30)`: agrega KPIs + daily avg + top anomalies + ranking por viatura.
- `vehicleFuelDetail(plate, days=60)`: histórico CANBUS + fills + anomalias da viatura.
- Baseline por viatura: média de L/100km dos últimos 30 dias (window móvel) — calculada em-memória a partir das rows (evita SQL complexo; ~60 viaturas × 30 dias = barato).

### Fase 2 · Dashboard `/fuel`

- Server component. `requireRole(["admin", "clarice"])`.
- `<PageHeader>`, 4 cards KPI, componente client `<FuelDailyChart>` (recharts), 2 `<table class="data-table">`.
- Botão `<form action={exportMonthlyReport}>` com ano/mês no form data.

### Fase 3 · Drill-down `/fuel/[plate]`

- Server component. Carrega viatura + 60 dias CANBUS + fills + anomalias.
- Componente client `<VehicleFuelChart>` com `ComposedChart` (bar fills + line canbus).
- Lista abastecimentos com StatusPill por fonte (Cepsa/Repsol/Radius Velocity/bomba interna).
- Para cada anomalia `open`: form com textarea motivo + botão `resolveAnomaly`.
- Para cada anomalia `resolved`: botão `reopenAnomaly`.

### Fase 4 · Server actions

- `resolveAnomaly(formData)` — id + reason obrigatório; update estado + audit.
- `reopenAnomaly(formData)` — id + reason obrigatório; update estado + audit.
- `exportMonthlyReport(formData)` — year/month; gera CSV; retorna string via redirect a /api/fuel/report ou (mais simples) devolve CSV como resposta. Alternativa adoptada: `exportMonthlyReport` retorna `{ filename, csv }` e o caller usa `<a download>` gerado no server. Implementação simples: action grava CSV em variável e retorna data URI via `redirect()` — desvio do padrão. Decisão final: a action faz o cálculo, devolve CSV como `string`, página recebe-o via hidden form que abre em nova tab com content-type text/csv. Para MVP, implementada como rota API `/api/fuel/report` chamada do botão — **mas** a instrução diz "server action → CSV". Compromisso: server action grava ficheiro temporário + retorna redirect — ou retorna CSV inline via headers. Mais pragmático no Next 15: action devolve string, botão chama via fetch, cliente faz download. Escolha: **função server exportada (não action)** chamada por rota GET `/api/fuel/report?year&month`, mas o enunciado restringe ficheiros a `actions.ts`. Compromisso final: action retorna `{ csv, filename }` e usamos `redirect()` não é possível com Blob. **Implementação pragmática**: `exportMonthlyReport` é server action que escreve `audit_log` e retorna `{ filename, csvContent }`. Cliente intermediário (pequeno wrapper `<ExportButton>`) consome via `useFormState` e dispara download no browser. Isto respeita "server actions" + "CSV download".
- Todas auditam: `fuel.anomaly_resolve`, `fuel.anomaly_reopen`, `fuel.export_report`.

## Riscos & decisões

- **Baseline inicial**: usei mediana móvel simples para evitar outliers puxarem baseline para cima. Código: `baseline = median(last30d.map(d => d.lPer100))`. Se frota tiver poucos dados (nova viatura), baseline = constante fallback 35 L/100.
- **Viaturas sem CANBUS** (~15%): excluídas do gráfico CANBUS + ranking L/100km (não há km fiável sem CANBUS); **incluídas** nos totais litros/EUR (via fills). Marcadas no ranking como "sem CANBUS".
- **CSV encoding**: UTF-8 com BOM (`\ufeff`) para Excel PT reconhecer acentos.
- **Recharts em client**: wrapper `components/platform/fuel-chart.tsx` com `"use client"`. Passamos dados já agregados do server.
